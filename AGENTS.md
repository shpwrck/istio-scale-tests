Use this file with Cursor agents working in this repository. For human-oriented setup and commands, prefer `README.md`.

## Source of truth

**Canonical procedures and command patterns** for multi-cluster Service Mesh on OpenShift are:

**[Red Hat OpenShift Service Mesh 3.3 ‚Äî Installing ‚Äî Multi-cluster topologies](https://docs.redhat.com/en/documentation/red_hat_openshift_service_mesh/3.3/html/installing/ossm-multi-cluster-topologies)**

## Implementation rules

- **No secrets in git:** Never add commits that contain secrets or credentials (see **Conventions for edits**).
- **Shell only:** Mesh setup and repo automation use `bash` under `istio-setup/` and `isotope-multicluster/`, and optionally under `terraform/**/scripts/` when that directory exists. Do not add Python, Ruby, or other runtimes for install or wiring unless the user explicitly requests it.
- **Numbered bash scripts:** Name repo-owned executable bash helpers `NNN-kebab-case.sh` with a **three-digit** prefix (`001`, `002`, ‚Ä¶). The number reflects **typical execution order** within that directory (same idea as `istio-setup/001`‚Äì`010`). Use this pattern for **all** such scripts in `istio-setup/`, `isotope-multicluster/`, `terraform/**/scripts/` (when used), and future automation directories‚Äînever add unnumbered `*.sh` peers without renumbering the folder. When you add or renumber a script, update **callers** (e.g. Terraform `external`, other bash wrappers) and **README / AGENTS** references in the same change.
- **Templates:** Prefer `manifests/ossm-multi-cluster/templates/*.yaml.tpl` with `envsubst` and variables from `config/versions.env` instead of duplicating per-cluster static YAML. Shared non-templated snippets may remain under `east-west/common/` when appropriate.
- **Avoid storing file content in scripts.** Use templates appropriate for the situation (Helm charts under `charts/`, `manifests/.../*.yaml.tpl` + `envsubst`, etc.) rather than large inline YAML or kubeconfig bodies in bash.
- **--dry-run:** Setup scripts that mutate clusters (`oc` / `kubectl` / `istioctl apply`) should accept `--dry-run` (typically `oc apply --dry-run=client`) so operators can validate renders without changing the cluster. Purely read-only scripts may treat `--dry-run` as documented no-op.
- **Pinned versions:** Maintain a single pin list in `config/versions.env`. Current targets: OpenShift 4.21.11, Kubernetes v1.34.6, Istio / Sail `spec.version` v1.28.5 (match `istioctl`), RHACM hub `ACM_CHANNEL` default `release-2.16` (supported with OpenShift 4.21.x per RHACM matrix ‚Äî bump with `OPENSHIFT_VERSION`). Bump `README.md` when pins change.
- **Script variables:** Expose and name env vars, flags, and bash identifiers consistently across `istio-setup/` ‚Äî see **Script variables and naming** (especially `SETUP_CONTEXTS`, `--contexts`, `CONTEXTS` / `DRY_RUN`).

## Script variables and naming (bash)

Keep `istio-setup/` consistent so operators can rely on the same env vars, flags, and internal names across scripts.

- **Defaults:** Scripts that use pinned versions or mesh-wide settings `source "${ROOT}/config/versions.env"` after setting `ROOT`. Put shared defaults (cluster lists, mesh/network IDs) in `config/versions.env` instead of copying literals into each script.
- **Kubernetes contexts ‚Äî environment:** `SETUP_CONTEXTS` ‚Äî comma-separated `kubectl` / `oc` context names (must match kubeconfig). Exported from `config/versions.env`. These names usually double as Istio primary-remote identifiers when they match `istioctl create-remote-secret --name`.
- **Kubernetes contexts ‚Äî CLI:** Where a script accepts multiple contexts, expose `--contexts CSV` using the **same** comma-separated format as `SETUP_CONTEXTS`. Scripts that take contexts positionally must still document defaults tied to `SETUP_CONTEXTS` in `usage` (and an `Environment:` block listing `SETUP_CONTEXTS` and any script-specific vars).
- **Internal names:** Parse `--contexts` into `CONTEXTS_CSV` (string), then split into a bash array `CONTEXTS[@]`. Loop with `ctx` when calling `oc --context="$ctx"` / `kubectl --context="$ctx"`. Prefer `CONTEXTS` for these arrays in **new or substantially edited** scripts; some older scripts use `CLUSTERS=(...)` ‚Äî treat as the **same** meaning (kube context names, not AWS/GCP cluster IDs).
- **Dry run:** Internal `DRY_RUN` (`0`/`1`); user-facing `--dry-run`. Pair them consistently and mention both in `usage` when applicable.
- **Repo root:** From `istio-setup/`, `ROOT="$(cd "$(dirname "$0")/.." && pwd)"`. Use the same depth pattern if you add scripts under subdirectories of `istio-setup/` (adjust `..` segments).
- **Templates (`envsubst`):** Per-context exports must stay aligned with template names already in `manifests/ossm-multi-cluster/templates/` (e.g. `CLUSTER_KEY`, `NETWORK` / `NETWORK_SUFFIX` from `versions.env`).
- **Per-context AWS env overrides:** Follow `config/versions.env`: variables such as `AWS_LOAD_BALANCER_*_<CONTEXT_SUFFIX>` where `CONTEXT_SUFFIX` is the kube context name uppercased with `/`, `:`, `-` replaced by `_` (example: `rosa-001` ‚Üí `ROSA_001`).
- **Exception ‚Äî 002-ossm-mc-cacerts.sh:** Uses `--clusters CSV` for **logical cluster keys** (directory names under `cacerts/`) and optional `--context-map key:ctx,...` when keys differ from kube context names. When keys match context names, keep `--clusters` aligned with `SETUP_CONTEXTS`.

## Purpose

This repository exists so operators can **end-to-end**: provision **ROSA** clusters (or equivalent OpenShift targets), **install multi-cluster Istio / OSSM** on them following **[Red Hat OpenShift Service Mesh multi-cluster documentation](https://docs.redhat.com/en/documentation/red_hat_openshift_service_mesh/3.3/html/installing/ossm-multi-cluster-topologies)** (multi-primary, multi-network meshes using the **Sail operator** ‚Äî `Istio`, `IstioCNI`), **load dynamic scale tests** into those clusters, and **produce reports** from the runs. Automation and manifests focus on that workflow rather than ad-hoc manual installs only.

## Conventions for edits

- **Shell:** Bash 4+; `set -euo pipefail` where already used; match existing style in `istio-setup/`.
- **Contexts:** Follow **Script variables and naming**; `rosa-001`, `rosa-002`, `rosa-003` are placeholders‚Äîoverride via `SETUP_CONTEXTS` / `--contexts`.
- **Paths:** Helpers resolve repo root via `"$(cd "$(dirname "$0")/.." && pwd)"` from `istio-setup/` or peer dirs (`isotope-multicluster/`, etc.) ‚Äî adjust `..` depth for deeper subtrees; keep this pattern for new automation scripts.
- **Secrets ‚Äî never commit:** Do **not** commit secrets, credentials, kubeconfigs with live tokens, private keys, CA material, API keys, or other sensitive values to git. Rely on `.gitignore` (e.g. `**/cacerts/`, `**/manifests/`) and local secret stores; use placeholders or templates for examples. If something might be secret, treat it as secret.
- **READMEs:** After each update, check **each** relevant README (`README.md` at repo root and under affected subtrees‚Äîe.g. `istio-setup/`, `manifests/`, `terraform/`) for necessary changes so commands, paths, prerequisites, and examples stay accurate. When you add or rename setup scripts, change defaults, or move YAML, update those READMEs in the same change.
