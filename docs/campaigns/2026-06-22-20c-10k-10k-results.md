## Per-suite findings

### Control plane — *the proof* (PASS, reproduced)
10,000 Services / 10,000 endpoints, namespace-scoped tuned baseline. **Run 1: 27%** peak util,
n_valid 60/60. **Run 2: 28%**, 60/60. Run-to-run variance **1 percentage point**. Zero istiod
restarts, zero Pending pods. (The 100/300-service ladder and `none`/`explicit` scopings were
budget-traded; the peak-×2 is the plan's stated minimum proof.)

### Data plane — cross-cluster latency (reproduced; partial coverage)
Single-source (cluster-002 → 19 remotes), fortio, QPS 10/100/500/1000 × 30s, 8 connections.
Cross-cluster (remote, via east-west gateway), aggregated over valid remotes — **reproduced across
two runs**:

| QPS | p50 | p90 | p99 | pct_200 |
|---|---|---|---|---|
| 10 | 4.33 ms | 4.88 ms | 6.76 ms | 100% |
| 100 | 3.63 ms | 4.20 ms | 4.92 ms | 100% |
| 500 | 3.28 ms | 3.87 ms | 4.69 ms | 100% |
| 1000 | 3.04 ms | 3.80 ms | 5.52 ms | 100% |

Local (same-cluster) p50 ~2.0ms; the east-west hop adds ~1.5–2.5ms. **Caveats:** (1) coverage was
**12/19 remotes** both runs — 7 specific clusters (007–012, 014) `FAILED` (see *Data-plane coverage*
below); (2) the local-baseline high-QPS rows tripped `ERROR_RATE_HIGH` — a **single-server-replica
saturation** artifact (sustained ~686/1000 QPS), not a mesh issue; (3) single-source, not all-pairs.

### Propagation — config freshness (PASS; harness fix required)
30 iterations, wall-clock P1/P2/P3 (the trustworthy convergence signal; histogram convergence
is non-load-bearing): **P1 local xDS ~2.2s · P2 remote EDS ~2.3s · P3 remote sidecar ~4.7s median (~5.8s mean).**
End-to-end mesh freshness at 20c/10k is **~4.7s median** — control-plane stages ~2.3s, the full data-plane
apply (P3) is the longer tail (~29% of changes served in <3s). (First attempt ground on a serial drain-wait bug — fixed,
PR #82; the clean 30-iter run gave the numbers above.)

### Churn — control plane under change (low-n; harness fix required)
Churn rate 5, scale 1→5. **Push amplification 1.0–1.1×** (one change → ~1.1 pushes mesh-wide — low,
good), local convergence **~7s** (consistent across n=5). A `grep -c` arithmetic-abort bug capped the
first run at n_valid=1 — fixed (PR #82); the re-run (stopped at n=5 for budget) confirmed the signal.

### Churn + data plane — traffic under churn (partial)
Churn rates 1/5/10, 60s baseline + 60s churn, QPS 200. **Rate 10/s: 600/600 ops, p50 2.99ms vs 2.82ms
baseline, Δp99 +1.72ms, 0 restarts → data-plane latency barely degrades under active config churn.**
Rate 1 churn phase OK (p50 2.93ms) but its baseline `FAILED` (no valid 1/s delta). Rate 5 churn phase +
Rate 1 baseline `FAILED` (the cross-cluster-7 data-plane issue bleeding in) — only 10/s yielded a complete pair.

## What this proves — and what it does not

**Proves:** 20 mesh-member clusters, 10,000 Kubernetes Service objects, 10,000 endpoints (1/Service)
formed one multi-primary mesh and operated within the stated control-plane headroom (≤28% SLA band,
19% istiod RSS), reproducibly; cross-cluster latency ~3–4ms; freshness ~4.7s median; low push amplification;
latency stable under churn.

**Does NOT prove:** multi-endpoint-per-Service fan-out (this run is 1 endpoint/Service — a 10×-endpoint
variant is ~10× the data-plane footprint and would not fit the 8-node pools); 10,000 globally-unique
hostnames (names repeat across clusters); all-pairs N×N data-plane throughput (single-source); behavior
under a fully-loaded simultaneous data plane (the non-control-plane suites are isolated baselines).

## Findings & fixes (codified)

- **BLOCKER #1 (ArgoCD↔ACM OpenAPI) — resolved correctly.** The codified "disable `server-foundation`"
  fix is *wrong*: `server-foundation` owns the managedcluster-import-controller, so disabling it breaks
  all spoke imports. MCE **2.11.3** is the genuinely-fixed build (the `cluster-lifecycle-api`
  UserPermission OpenAPI bug; upstream stolostron #56/#57, ACM-35283). We **enabled** server-foundation
  on MCE 2.11.3, verified `/openapi/v2` parses clean, and the hub app-of-apps synced (the stress test
  never reached this). **GO action:** require MCE ≥ 2.11.3; disable nothing.
- **PR #80** — gateway HPA pin (`autoscaling.enabled` toggle); `--qps/--burst` flag-support detection
  (both `oc` 4.21 and kubectl v1.34 reject them → silently broke every call); platform `set_list`
  empty-list `length()` guard.
- **PR #81** — gateway `replicaCount` pin (deterministic gateways under traffic).
- **PR #82** — propagation drain parallel+bounded (`PROPAGATION_DRAIN_TIMEOUT_SEC`); churn baseline
  `grep -c || echo 0` → `0\n0` arithmetic-abort fix + per-iteration resilience.
- **AWS Classic-ELB quota** (raised 20→50): 40 gateway LoadBalancer Services (Classic ELBs) exceed the
  default per-region quota of 20. **A real infra-quota gate** alongside EIP/NAT/vCPU — add a preflight.
- **Data-plane coverage (12/19)** — root cause is **data-plane east-west-gateway traffic**, NOT
  control-plane: the 7 failing clusters (007–012, 014) show `istioctl remote-clusters` **synced** and
  healthy gateways/LBs, yet cross-cluster mTLS through their NLBs fails. They are exactly the clusters
  whose east-west LBs provisioned *late* (after the Classic-ELB quota raise) + were churned (HPA pin) —
  a residual NLB target-registration/passthrough issue. Fix = restart the east-west gateway pods /
  re-register NLB targets (not istiod). Left unconfirmed at teardown (out of budget).
