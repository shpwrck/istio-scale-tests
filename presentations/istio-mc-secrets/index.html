Istio Multicluster on OpenShift

cert-manager + External Secrets + ACM · the secrets that wire the mesh together

A 20-minute walk through the dependency graph

Welcome. In the next 20 minutes I will show you how a multi-primary, multi-network Istio service mesh that spans two or more OpenShift clusters is actually wired up — focusing on the three Kubernetes Secrets that make it work and the four operators (cert-manager, External Secrets, Argo CD, and ACM) that produce and distribute those secrets. The audience I have in mind is fluent in Kubernetes, Helm, and RBAC but new to Istio multicluster, so I will define every Istio-specific term the first time it appears.

The problem

  • Two (or more) OpenShift clusters, possibly in different networks.
  • One logical service mesh — workloads on cluster A can call services on cluster B by their normal Kubernetes Service name.
  • mTLS everywhere; cross-cluster traffic stays encrypted end-to-end.

The traffic path is not the hard part — gateways, routes, and Envoy already know how to do that. The hard part is secret distribution: three different Kubernetes Secrets must exist on each cluster, with the right contents, signed by the right authority, before the mesh works at all.

The runtime story — Envoy routing traffic to another cluster — is well documented. What trips teams up in practice is getting the prerequisite secrets onto the right cluster, in the right namespace, with the right label, signed by a shared CA. That's the chain we are going to walk today.

Vocabulary (defined once, used everywhere)

  • Mesh ID — a logical identifier shared by all clusters in one mesh. Same value on every cluster. Example: mesh1.
  • Cluster name — Istio's per-cluster label (distinct from the Kubernetes cluster name in kubeconfigs). Example: cluster-a.
  • Network ID — the L3/L4 reachability domain. Clusters with different network IDs cannot reach each other's pod IPs directly — traffic must go through an east-west gateway.
  • Trust domain — the SPIFFE root of identity. All clusters in one mesh share one trust domain — that is what lets cluster A's mTLS certificate be valid on cluster B.
  • East-west gateway — a special Istio gateway listening on port 15443 with TLS mode AUTO_PASSTHROUGH. It forwards mTLS traffic between networks without terminating TLS.
  • Remote secret — a kubeconfig-shaped Kubernetes Secret in istio-system, labeled istio/multiCluster: true. It tells istiod how to read another cluster's Kubernetes API.
Six terms. We will see all of them in the next thirteen minutes. Two key non-obvious facts: one, trust domain is what allows a workload on cluster A to verify a workload identity on cluster B cryptographically — it's a SPIFFE concept, not a Kubernetes one. Two, AUTO_PASSTHROUGH on the east-west gateway means the gateway pod never holds the inner mTLS keys — it just SNI-routes encrypted bytes onward. That is how the gateway can be operated by a different team than the workloads it carries.

Topology: multi-primary, multi-network

Two clusters, two networks. Each cluster runs its own istiod — that's what "multi-primary" means: there is no central control plane, every cluster is self-sufficient. The east-west gateway listens on 15443 with TLS mode AUTO_PASSTHROUGH: it reads the inner mTLS SNI and forwards the encrypted bytes without unwrapping them. The three Secrets that make this work

Secret Lives on Purpose
cacerts each spoke istio-system intermediate CA istiod uses to sign workload certificates
kubeconfig-{cluster} hub external-secrets-operator ns how the hub talks to each spoke apiserver (used by ESO)
remote secret (istio/multiCluster: true) each spoke istio-system tells istiod on one spoke how to read the other spoke's apiserver

The next slides cover each of these — what it must contain, who produces it, and how it lands on the right cluster.

Three secrets. The first one — cacerts — is the cryptographic root: every workload identity in the mesh chains up to a CA whose intermediate lives in this Secret. The second — kubeconfig-{cluster} — exists on the hub and gives the hub apiserver-level access into each spoke; ESO needs this to push things in. The third — the remote secret — also lives on each spoke, but it's the OTHER spoke's kubeconfig, so istiod can watch services across the boundary. We're going to follow them in cryptographic-root-first order: cacerts, then remote secrets, then how they all get there.

cacerts — the four keys

A Kubernetes Secret named cacerts in istio-system, with four data keys:

  • ca-cert.pem — the per-cluster intermediate CA cert. istiod presents this when signing workload certs.
  • ca-key.pem — the private key for that intermediate. This is the high-value secret.
  • root-cert.pem — the shared root CA cert. Same bytes on every cluster in the mesh.
  • cert-chain.pem — intermediate + root concatenated (the chain workloads need to verify peers).

Why a shared root? Because mTLS between a workload on cluster A and a workload on cluster B requires that both peers' certificates validate against a common chain. The intermediate can (and does) differ per cluster; the root is identical everywhere.

The shape of cacerts is fixed by Istio — istiod will look for these exact four file names inside the Secret. If you produce them with cert-manager you have to assemble the four keys yourself; cert-manager Certificate resources don't produce a Secret in this shape by default. We'll see the assembly in the next slide. The intermediate's private key is the highest-value secret in this whole stack — never put it in git, never expose it to a spoke that doesn't need it. Each spoke gets its own intermediate so that revoking one cluster doesn't compromise the others.

cert-manager assembles cacerts

Step one: on the hub, a self-signed Certificate is the root, paired with a ClusterIssuer of the same name. Step two: one intermediate Certificate per spoke is issued by that ClusterIssuer — both intermediates roll up to the same root. Step three: cert-manager produces a Kubernetes Secret with the standard TLS shape (tls.crt, tls.key, ca.crt) — but istiod wants four very specific keys. An ESO ExternalSecret reshapes the cert-manager Secret into the four-key cacerts shape, using its v2 templating engine. Worth highlighting: cert-chain.pem is just the intermediate cert concatenated with the root cert. Crucially the assembled cacerts still lives on the hub at this point — getting it onto the spoke is ESO's PushSecret job, two slides from now.

The remote secret — what istiod actually reads

apiVersion: v1
kind: Secret
metadata:
  name: istio-remote-secret-cluster-b   # the OTHER cluster's name
  namespace: istio-system                # always
  labels:
    istio/multiCluster: "true"           # this label is the trigger
data:
  cluster-b: <base64 kubeconfig pointing at cluster-b's apiserver>
  • The key inside data is the remote cluster's name; the value is a full kubeconfig.
  • The istio/multiCluster: true label is what istiod watches for — without it the Secret is ignored.
  • The kubeconfig's ServiceAccount on the remote cluster needs read access to Services, EndpointSlices, Pods, and Nodes.
This is the actual Secret shape. The Istio convention is to name it istio-remote-secret-{remote-cluster}. The data key matches the remote cluster name; the data value is the full kubeconfig content. The label istio/multiCluster=true is what istiod's secret controller watches for — drop the label and the cluster silently disappears from the mesh. The kubeconfig's identity does not need cluster-admin: read access on Services, EndpointSlices, Pods, and Nodes is sufficient. ACM's ManagedServiceAccount creates exactly such a scoped identity, which is one reason this stack uses it.

How the remote secret drives discovery

Step 1: the remote secret sits in cluster-a's istio-system, labeled istio/multiCluster=true. Step 2: istiod on cluster-a — specifically its multicluster controller — reads the Secret. Step 3: using the kubeconfig in the Secret, istiod-A opens a watch against cluster-b's kube-apiserver for Services, EndpointSlices, Pods, and Nodes. Step 4: those endpoints flow back into istiod-A and get pushed via xDS to every sidecar on cluster-a, with the east-west gateway as the next-hop address — which closes the loop with slide four. Symmetric on the other side: cluster-b's istio-system has an istio-remote-secret-cluster-a doing the inverse.

So who puts these Secrets on the spokes?

  • cacerts exists on the hub (cert-manager produced it there) — but istiod needs it on each spoke.
  • The remote secrets are derived on the hub (one kubeconfig per spoke) — but each spoke needs the other spoke's secret.
  • You do not want to write a bash loop with kubectl --context.

We use the External Secrets Operator (ESO) — specifically its PushSecret resource — to ship the assembled Secrets from the hub into each spoke's istio-system namespace.

Up to this point everything lives on the hub. We need to fan it out. We pick ESO PushSecret rather than Argo CD because the data is sensitive — we want the source of truth to stay on the hub (in real Secrets, not git), and have a small operator copy it to the spokes. ESO's PushSecret resource does exactly that direction: hub Secret as source, spoke Secret as destination.

ESO PushSecret: hub → each spoke

Two source-Secret kinds on the hub: cacerts-X (assembled from cert-manager output, as we saw in the previous animation) and kubeconfig-X (an actual usable kubeconfig — we'll see how it's produced two slides from now). ESO has a SecretStore per spoke, authenticated using kubeconfig-X. Then two kinds of PushSecret: push-cacerts-X copies cacerts-X to spoke X as "cacerts". push-remote-secret-X is more interesting — it reads kubeconfig-X but uses the v2 template feature to materialize a Secret named istio-remote-secret-X on the OTHER spokes, with label istio/multiCluster=true applied at template time. So kubeconfig-cluster-a crosses over to cluster-b, and vice-versa. PushSecret direction is always hub-source to spoke-target — inverse of ExternalSecret which reads from a remote source.

Upstream of ESO: where kubeconfig-cluster-a comes from

Step 1: ACM ManagedCluster CR is applied on the hub. The klusterlet on the spoke does the actual registration. Step 2: ACM ManagedServiceAccount (an ACM addon) creates a real ServiceAccount on the spoke and mints a bearer token, then mirrors the token Secret back to the hub. The ServiceAccount itself has no RBAC by default — a ClusterRoleBinding has to be applied separately to give it the needed permissions on the spoke. Step 3: a GitOpsCluster CR plus an ACM Placement tell Argo CD which managed clusters to register. For each one Argo CD writes a cluster Secret in openshift-gitops named {cluster}-application-manager-cluster-secret. Step 4: a small ESO ExternalSecret — defined in chart hub-kubeconfig-from-argosecret in this repo — reads the server URL and bearer token from that cluster Secret and templates a full kubeconfig into a Secret named kubeconfig-{cluster} on the hub. That last Secret is what slide 11 used to authenticate into the spokes. The full dependency chain is closed.

What breaks, and how you'd see it

Symptom Likely cause Check
cross-cluster mTLS handshake fails cacerts root differs between clusters diff <(oc --context A get secret cacerts -n istio-system -o jsonpath='{.data.root-cert\.pem}') ...
cluster B services not visible on cluster A remote secret missing or unlabeled istioctl remote-clusters --context A
istiod sees cluster B but endpoints are empty kubeconfig token expired or RBAC too narrow istioctl proxy-config endpoints <pod> --context A
east-west gateway returns 503 workload's mesh ID or trust domain doesn't match peer compare Istio CR fields on both clusters
Four most common failure modes. The first three are secret-distribution problems; the last is a config-drift problem. The single most useful command in this entire stack is istioctl remote-clusters — it shows you which remote secrets istiod has loaded and the connection status. If you see no remote clusters, your label is wrong or your Secret is in the wrong namespace.

Recap · the dependency order

  1. cert-manager on the hub builds one shared root CA and one intermediate CA per spoke.
  2. ACM registers each spoke and ManagedServiceAccount mints a scoped token.
  3. Argo CD writes a cluster Secret per spoke; ESO ExternalSecret reshapes it into a kubeconfig.
  4. ESO PushSecret uses those kubeconfigs to ship cacerts + istio-remote-secret to each spoke's istio-system.
  5. istiod on each spoke reads its remote secret and starts watching the other apiserver — the mesh is live.
