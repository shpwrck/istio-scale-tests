Istio Multicluster on OpenShift

cert-manager + External Secrets + ACM · the secrets that wire the mesh together

A 20-minute walk through the dependency graph

Welcome. In ~20 minutes we'll walk through how a multi-primary, multi-network Istio mesh spanning two OpenShift clusters is wired up — the three Kubernetes Secrets that make it work and the four operators (cert-manager, External Secrets, Argo CD, ACM) that produce and distribute them. Audience: fluent in Kubernetes/Helm/RBAC but new to Istio multicluster, so Istio terms get a one-line gloss on first use.

The problem

  • Two (or more) OpenShift clusters, possibly in different networks.
  • One logical service mesh — workloads on cluster A can call services on cluster B by their normal Kubernetes Service name.
  • mTLS everywhere; cross-cluster traffic stays encrypted end-to-end.

The traffic path is not the hard part — gateways, routes, and Envoy already know how to do that. The hard part is secret distribution: three different Kubernetes Secrets must exist on each cluster, with the right contents, signed by the right authority, before the mesh works at all.

The runtime story — Envoy routing traffic to another cluster — is well documented. What trips teams up in practice is getting the prerequisite secrets onto the right cluster, in the right namespace, with the right label, signed by a shared CA. That's the chain we are going to walk today.

Vocabulary (defined once, used everywhere)

  • Network ID — the L3/L4 reachability domain. Clusters with different network IDs cannot reach each other's pod IPs directly — traffic must go through an east-west gateway.
  • Trust domain — the SPIFFE root of identity. All clusters in one mesh share one trust domain — that is what lets cluster A's mTLS certificate be valid on cluster B.
  • East-west gateway — a special Istio gateway listening on port 15443 with TLS mode AUTO_PASSTHROUGH. It forwards mTLS traffic between networks without terminating TLS.
  • Remote secret — a kubeconfig-shaped Kubernetes Secret in istio-system, labeled istio/multiCluster: true. It tells istiod how to read another cluster's Kubernetes API.
Four terms — others (mesh ID, ManagedCluster, etc.) get inline glosses on first use later. Two non-obvious facts: trust domain lets cluster A verify cluster B identities cryptographically — a SPIFFE concept, not Kubernetes. AUTO_PASSTHROUGH means the east-west gateway pod never holds inner mTLS keys; it SNI-routes encrypted bytes onward, so a different team can operate it from the workloads it carries.

Multi-primary, multi-network topology

istiod control plane workload + sidecar east-west gateway cross-cluster call
Two clusters, two networks. Each cluster runs its own istiod — that is what "multi-primary" means: no central control plane, every cluster self-sufficient. The east-west gateway listens on 15443 with TLS mode AUTO_PASSTHROUGH: it reads the inner mTLS SNI and forwards the encrypted bytes without unwrapping them. Click to show the cross-cluster service call: the source sidecar dials the REMOTE east-west gateway directly across the network boundary. The call traverses the remote east-west gateway (EW gw B for an A→B call) — the local EW gateway only carries traffic arriving from other networks (ingress side). Trust domain and mesh ID are both mesh1 on both clusters (mesh ID — Istio's name for the mesh as a whole; one string shared by all member clusters); we will revisit those when we get to cacerts.

The three Secrets that wire the mesh

Terminology: hub = ACM management cluster; spokes = managed clusters where workloads run.

Secret Lives on Purpose
cacerts each spoke istio-system intermediate CA istiod uses to sign workload certificates
remote secret (istio/multiCluster: true) each spoke istio-system tells istiod on one spoke how to read the other spoke's apiserver
kubeconfig-{cluster} hub only
(ESO namespace)		 how the hub authenticates into each spoke (used by ESO to ship the two Secrets above)
Three secrets. cacerts is the cryptographic root: every workload identity in the mesh chains up to a CA whose intermediate lives in this Secret. The remote secret also lives on each spoke, but it's the OTHER spoke's kubeconfig — it tells istiod how to watch services across the boundary. kubeconfig-{cluster} is the third, and the only one that does NOT live on a spoke — it lives only on the hub, and ESO uses it as the auth credential to ship the first two Secrets into the spokes. We'll follow them in this order: cacerts, then the remote secret on the spoke, then ESO's distribution, then the upstream of distribution.

cacerts — the four keys

A Kubernetes Secret named cacerts in istio-system, with four data keys:

  • ca-cert.pem — the per-cluster intermediate CA cert. istiod presents this when signing workload certs.
  • ca-key.pem — the private key for that intermediate. This is the high-value secret.
  • root-cert.pem — the shared root CA cert. Same bytes on every cluster in the mesh.
  • cert-chain.pem — intermediate + root concatenated (the chain workloads need to verify peers).

Ordering matters: cacerts must exist on the spoke before istiod first starts — otherwise istiod self-signs its own CA and the cluster can never join the mesh later without redeploying.

The shape of cacerts is fixed by Istio — istiod looks for these exact four file names. cert-manager Certificates don't produce a Secret in this shape by default, so we assemble it (next slide). The intermediate's private key is the highest-value secret in the stack. Each spoke gets its own intermediate, so revoking one cluster doesn't compromise the others. Shared root is what lets cluster A's mTLS validate against cluster B's chain.

cert-manager assembles cacerts

Step one: a bootstrap selfSigned ClusterIssuer (mesh-selfsigned-bootstrap) signs the root Certificate (mesh-root-ca, isCA: true). The root Certificate's resulting Kubernetes Secret backs a CA-mode ClusterIssuer of the same name via spec.ca.secretName — that is the issuer downstream resources reference. Step two: one intermediate Certificate per spoke is issued by mesh-root-ca, all rolling up to the same root (only one box drawn to keep the diagram compact). Step three: cert-manager produces a Kubernetes Secret with the standard TLS shape (tls.crt, tls.key, ca.crt) but istiod wants four very specific keys, so an ESO ExternalSecret reshapes it into the four-key cacerts Secret using its v2 templating engine. cert-chain.pem is just the intermediate concatenated with the root. The assembled cacerts still lives on the hub at this point — getting it onto the spoke is ESO's PushSecret job, on the PushSecret distribution slide later.

The remote secret — what istiod reads

apiVersion: v1
kind: Secret
metadata:
  name: istio-remote-secret-cluster-b   # the OTHER cluster
  namespace: istio-system               # always
  labels:
    istio/multiCluster: "true"          # the trigger label
data:
  cluster-b: <base64 kubeconfig for cluster-b's apiserver>
  • The key inside data is the remote cluster's name; the value is a full kubeconfig.
  • The istio/multiCluster: true label is what istiod watches for — without it the Secret is ignored.
  • The kubeconfig's ServiceAccount on the remote cluster needs read access to Services, EndpointSlices, Pods, and Nodes.
This is the actual Secret shape. The Istio convention is to name it istio-remote-secret-{remote-cluster}. The data key matches the remote cluster name; the data value is the full kubeconfig content. The label istio/multiCluster=true is what istiod's secret controller watches for — drop the label and the cluster silently disappears from the mesh. The kubeconfig's identity does not need cluster-admin: read access on Services, EndpointSlices, Pods, and Nodes is sufficient. ACM's ManagedServiceAccount creates exactly such a scoped identity, which is one reason this stack uses it.

How the remote secret drives discovery

Step 1: the remote secret sits in cluster-a's istio-system, labeled istio/multiCluster=true. Step 2: istiod on cluster-a — specifically its multicluster controller — reads the Secret. Step 3: using the kubeconfig in the Secret, istiod-A opens a watch against cluster-b's kube-apiserver for Services, EndpointSlices, Pods, and Nodes. Step 4: those endpoints flow back into istiod-A and get pushed via xDS to every sidecar on cluster-a, with the east-west gateway as the next-hop address — which closes the loop with slide four. Symmetric on the other side: cluster-b's istio-system has an istio-remote-secret-cluster-a doing the inverse.

ESO PushSecret: hub → spokes

Up to this point everything lives on the hub; ESO PushSecret fans it out to the spokes. (We pick ESO PushSecret over Argo CD here because the data is sensitive — source of truth stays in real Secrets on the hub, never git, and a small operator copies it to each spoke.) Two source-Secret kinds on the hub: cacerts-X (assembled from cert-manager output, prior animation) and kubeconfig-X (we'll see how it's produced next slide). ESO has a SecretStore per spoke authenticated using kubeconfig-X. Two kinds of PushSecret then run: push-cacerts-X copies cacerts-X to spoke X as "cacerts". push-remote-secret-X is more interesting — it reads kubeconfig-X but uses the v2 template feature to materialize a Secret named istio-remote-secret-X on the OTHER spoke, with label istio/multiCluster=true applied at template time. So kubeconfig-cluster-a crosses over to cluster-b, and vice-versa — the counter-intuitive bit worth pausing on. PushSecret direction is always hub-source to spoke-target — inverse of ExternalSecret. Reminder: cacerts must already be on the spoke before istiod starts there, or istiod self-signs and you cannot rejoin the mesh later without redeploying — so this PushSecret has to win the race with the spoke's istiod boot.

Building kubeconfig-cluster-a

Step 1: ACM ManagedCluster CR on the hub and the klusterlet on the spoke complete registration; then ACM ManagedServiceAccount (an addon) creates a ServiceAccount on the spoke, mints a bearer token, and mirrors the token Secret back to the hub. The ServiceAccount itself has no RBAC by default — a ClusterRoleBinding is applied separately on the spoke to grant the needed permissions. Step 2: a GitOpsCluster CR plus an ACM Placement tell Argo CD which managed clusters to register; for each one Argo CD writes a cluster Secret in openshift-gitops named {cluster}-application-manager-cluster-secret. Step 3: an ESO ExternalSecret (chart hub-kubeconfig-from-argosecret) reads the server URL, CA data, and bearer token (or client cert/key) from that cluster Secret and templates a full kubeconfig into Secret kubeconfig-{cluster} on the hub. That last Secret is what the previous slide's per-spoke SecretStore authenticated with. Dependency chain closed.

What breaks, and how you'd see it

Symptom Likely cause Check
cross-cluster mTLS handshake fails cacerts root differs between clusters diff <(oc --context A get secret cacerts -n istio-system -o jsonpath='{.data.root-cert\.pem}') ...
cluster B services not visible on cluster A remote secret missing or unlabeled istioctl remote-clusters --context A
istiod sees cluster B but endpoints are empty kubeconfig token expired or RBAC too narrow istioctl proxy-config endpoints <pod> --context A
east-west gateway returns 503 SNI cluster mismatch — workload not labeled into the right network, or remote EW gateway not advertising the expected cluster istioctl proxy-config clusters <ew-gw-pod> and verify topology.istio.io/network labels
Four most common failure modes. The first three are secret-distribution problems; the last is a config-drift problem. The single most useful command in this entire stack is istioctl remote-clusters — it shows you which remote secrets istiod has loaded and the connection status. If you see no remote clusters, your label is wrong or your Secret is in the wrong namespace.

Recap · the dependency order

  1. cert-manager on the hub builds one shared root CA and one intermediate CA per spoke.
  2. ACM registers each spoke and ManagedServiceAccount mints a scoped token.
  3. Argo CD writes a cluster Secret per spoke; ESO ExternalSecret reshapes it into a kubeconfig.
  4. ESO PushSecret uses those kubeconfigs to ship cacerts + istio-remote-secret to each spoke's istio-system.
  5. istiod on each spoke reads its remote secret and starts watching the other apiserver — the mesh is live.
